# RESEARCH.md — Why Windows Defender Interferes with Honeypot Executables

This explains the mechanism behind the "inconsistent hash on open" behavior seen
with `HelloWorld.exe`-style entries, and why it is specific to executables (not
`.pdf`, `.docx`, `.png`, etc.).

## 1. Background: minifilters and altitude

Windows file I/O passes through a stack of **minifilter drivers**, coordinated by
the **Filter Manager** (`fltmgr.sys`). Each minifilter registers at a numeric
**altitude**, allocated and tracked centrally by Microsoft. Altitude determines
stacking order:

> "A filter driver that has a low numerical altitude is loaded into the I/O
> stack below a filter driver that has a higher numerical value."
> — [Load order groups and altitudes for minifilter drivers, Microsoft Learn](https://learn.microsoft.com/en-us/windows-hardware/drivers/ifs/load-order-groups-and-altitudes-for-minifilter-drivers)

In plain terms: **higher altitude = closer to the application, lower altitude =
closer to the actual file system.** A single I/O request (open, read, write)
travels down through the stack from the highest altitude to the lowest, then
the result travels back up. Every minifilter attached to that volume gets a
chance to inspect or modify the request on the way down and the result on the
way up.

Altitudes are grouped into ranges by purpose (anti-virus, encryption,
replication, virtualization, etc.), published by Microsoft in the
[Allocated Filter Altitudes](https://github.com/MicrosoftDocs/windows-driver-docs/blob/staging/windows-driver-docs-pr/ifs/allocated-altitudes.md)
reference. Two entries matter here:

| Driver         | Altitude   | Load order group          | Role                          |
|----------------|-----------:|----------------------------|--------------------------------|
| `WdFilter.sys` | **328010** | FSFilter Anti-Virus (320000–329998) | Windows Defender real-time protection |
| `prjflt.sys`   | **189800** | FSFilter HSM (180000–189999)        | Windows Projected File System (ProjFS) — what PhantomFS is built on |

`328010 > 189800`, so **WdFilter sits well above prjflt in the stack.** This is
the load-order group design intentionally: anti-virus is meant to see fully
resolved file content, after any lower-level filter (virtualization, HSM,
compression, encryption) has produced the real bytes.

## 2. What that ordering means for PhantomFS specifically

When a process opens or reads a file under the honeypot's virtual root:

1. The request enters the stack at WdFilter's altitude first (it's higher).
2. It continues down through the stack until it reaches prjflt.
3. prjflt recognizes the path as a ProjFS placeholder and invokes **PhantomFS's
   own callbacks** — `GetPlaceholderInfo` / `GetFileData` — to materialize the
   file's content on demand. This is exactly the code in `PhantomFS.cs`
   (`OnGetPlaceholderInfo`, `OnGetFileData`, `SyntheticContent.Generate`).
4. The generated bytes flow back up through the stack. WdFilter, being above
   prjflt, sees this **already-hydrated, fully decoded content** — the same
   bytes any other reader would see. Defender is not bypassing PhantomFS's
   virtualization; architecturally, it can't — it's strictly downstream of it.

This matters for diagnosis: **PhantomFS's own decode pipeline is deterministic.**
Given a fixed `size:` and a fixed embedded payload, `SyntheticContent.Generate`
always returns the same bytes (the base64/gzip payload is decoded once and
cached; `FitToSize` is a pure function of that cached array and the declared
size). So if a hash of the served file changes between accesses, the change is
not happening inside PhantomFS — it's happening somewhere *above* it, after the
bytes leave the ProjFS layer.

## 3. Why executables specifically trigger this, and .pdf/.png don't

WdFilter's real-time, on-access scan is most aggressive for executable content
(PE files), for the well-known reason that malware most commonly arrives as an
executable. On `CreateFile`/open of an eligible file type, WdFilter typically:

- Intercepts the post-create callback (after the path has resolved all the way
  down through the stack, including through prjflt's hydration).
- Issues its **own internal read** of the file's content, back down through the
  same filter stack, to run signature and heuristic scanning — and, depending on
  configuration, submits data to cloud-delivered protection for a verdict.
- Holds the original caller's I/O pending until it reaches a verdict.
- If flagged, takes an action — typically quarantine (moving/renaming the file
  out of place) — which happens at the file-system level, entirely outside
  PhantomFS's own logic.

This has two consequences worth separating clearly:

- **A single logical "open" can trigger multiple `GetFileData` callbacks** —
  one for WdFilter's internal scan-read, and potentially another for the
  original caller's read once WdFilter releases it. You may see more than one
  alert fire for what looks like one access.
- **If Defender quarantines the file**, anything read afterward is no longer
  the bytes PhantomFS generated — it's whatever the quarantine action left
  behind (which can be a stub, an access-denied state, or nothing at all,
  depending on configuration). Hashing across that boundary will very reasonably
  produce different results on different attempts, even though PhantomFS's own
  output never changed.

This is the most likely explanation for what was observed, but it is an
inference from documented Defender/ProjFS architecture, not something directly
observed in your environment yet. `Watch-DefenderInterference.ps1` (below) is
built to confirm or rule this out concretely: it hashes the served file
repeatedly while simultaneously polling Defender's own detection/event data, so
a hash change can be directly correlated with a Defender action rather than
assumed.

## 4. Practical implication

This is exactly the protective behavior you'd want from a honeypot deployment —
Defender scanning and potentially quarantining an unknown, unsigned executable
appearing in an unexpected location is the system working as intended. It also
means: if you want to reliably test the base64gzip *decode path itself* end to
end (confirm PhantomFS serves the bytes you expect), either use a non-executable
decoy (`.pdf`, `.docx`, `.png` — anything not in the "PE file" class that
triggers on-access AV scanning), or temporarily add the honeypot's virtroot to
Defender's exclusion list for the duration of the test — see the script's
`-CheckExclusions` output for the exact current state.
